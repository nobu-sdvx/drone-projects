## 1. プロジェクト概要

応用機械設計製図Ⅰ「ドローン班」課題。屋内用の **固定翼型ドローン(総重量 40g 以下)** を 3 人チームで設計・製作。8 月上旬の競技会出場が最終ゴール。

- **次マイルストーン**: 2026-05-18 第 2 回審査会(概略計算書審査)
- **現フェーズ**: 計算書 §2 主翼の文書化、§3 尾翼設計の着手

詳細は以下を参照:
- `requirements.md` … 自己設定の設計目標と確定済み判断
- `official_requirements.md` … 課題 PDF と POWERUP 4.0 仕様から抽出した公式要件
- `progress_report_YYYY-MM-DD.md` … 日次進捗(最新を優先参照)

## 3. 未決事項の詳細

### 3.1 ⚠️ 失速速度要件 未達(最優先)

- **要件**: 4 m/s 以下 / **現状**: 5.5 m/s(40g時) → 1.5 m/s 乖離
- **物理的原因**: V_stall = √(2W/ρ·S·CL_max)。4 m/s 達成には CL_max=1.93 が必要だが、低 Re 翼型では実現不可能。
- **対策案(5/18 までに方針決定)**:

| 案 | 内容 | 影響 |
|---|---|---|
| A | 翼面積拡大 216→300 cm²(翼幅 360→500 mm) | 重量+0.8g、製作難度やや上昇 |
| B | 要件緩和 4→5 m/s 以下 | 要件定義書書き換え、審査会で説明 |
| C | 総重量厳守 40g→32.5g | 製作精度で重量管理を厳格化 |

詳細根拠:`progress_report_2026-04-23.md` §3.1

### 3.2 CFRP ロッドの正式材料データ未取得

- 現状、`mainwing_structural_analysis.py` 内の `SIGMA_ULT = 1500e6` Pa は**典型値**を使用
- 実購入予定品のデータシート値に**要更新**
- 影響範囲:翼根応力 SF の絶対値

### 3.3 φ2mm スパーの過剰設計可能性(軽量化候補)

| スパー直径 | 断面係数 W [mm³] | σ_design (n=2) | SF |
|---|---|---|---|
| 2.0 mm | 0.785 | 29 MPa | 52 |
| 1.5 mm | 0.331 | 69 MPa | 22 |
| 1.2 mm | 0.170 | 134 MPa | 11 |

- φ1.2mm でも SF=11 と十分。重量約 0.18g 削減可能。
- **ただし座屈・接合部応力集中は別評価が必要** → Phase 3 詳細設計 FEM 検証。
- 現時点では φ2mm を確定値とし、軽量化候補として記録のみ。

## 4. Claude Code への運用ルール ★最重要

このセクションは Nobu 個別の好み。**毎回これに従って動くこと。**

### 4.1 Git 操作は自由に行ってよい
- このリポジトリは **個人リポ**。ブランチ作成・コミット・マージは Claude Code の判断で実行 OK
- ただし、**実行前にひと言**:何を変更し、何のためのコミットなのかを簡潔に説明してから走らせる(後で履歴を辿れるように)
- コミットメッセージは具体的に(例:`mainwing_structural_analysis.py: NCrit 補正項を追加、α=3° で SF=48 に微減`)
- 大きな改変や設計判断に関わる変更(数値の確定値書き換え等)は、自動で進めず一度確認を取る

### 4.2 情報の根拠を必ず示す
- 数値・主張には出処を明示する(例:「片柳『飛行機設計入門』§X.X より」「XFLR5 出力ファイル `spanwise_*.txt` の N 行目より」)
- 不確かな情報は「未確認の推定」と明示する
- 計算結果を引用するときは、生成元のスクリプトとデータファイルを併記する

### 4.3 新しい概念には例え話を添える
- Nobu は API は何個か作った経験ありの初心者レベル
- 「要するに XX のような役割」「こう覚えておけばよい」というまとめ方を好む

### 4.4 ファイル変更時は「役割」を先に説明してから差分を出す
- いきなり diff を出さず、まず以下を伝える:
  1. **どのファイル**(例:`mainwing_structural_analysis.py`)
  2. **そのファイルの役割**(例:主翼スパーの曲げ応力と安全率を計算する)
  3. **どの関数・どの行**(例:`compute_bending_moment()` 関数)
  4. **何を追加・変更するのか**(例:NCrit 補正項を加える)
- そのあとで差分を提示

### 4.5 計算書・報告書からは AI 用語を消す
- 内部議論では「NeuralFoil で」「AI 解析で」と書いて OK
- ただし**提出資料**(計算書、審査会スライド)では「数値解析により」「XFLR5 により」など、ツール名は書くが「AI」というラベルは外す
- スタンスは「設計者がこの設計を一人称で保証する」

### 4.6 Google Drive のパスを決め打ちしない
- Claude Code は Drive に直接アクセスできない
- 図を生成するスクリプトは **ローカル `plots/` に出力**、ユーザーが手動で Drive にコピーする運用
- スクリプト内で Drive 側のフルパスを書かない

### 5.1 計算書の章と解析スクリプトの対応

各章の数値は専用スクリプトで算出する。実行は `venv/Scripts/python.exe` + 環境変数 `PYTHONIOENCODING=utf-8`(Windows の cp932 で日本語 print が落ちるため)。

| 章 | スクリプト | 役割 |
|---|---|---|
| §2 主翼 | `mainwing_structural_analysis.py` | VLM 生データ読出 → 巡航外挿 → 翼根曲げ・応力・SF |
| §3 尾翼 | `Tail_design_analysis.py` | トリム尾翼力・容積比・寸法・強度・静安定 |
| §4 胴体 | `2026-05-17_boom_attachment_analysis.py` | ブーム接合方式変更の重量・重心・強度影響 |
| §5 プロペラ | `propeller_design_analysis.py`(空力)/ `propeller_structural_analysis.py`(強度)/ 定数は `propeller_config.py` | QPROP 空力性能・遠心力・安全率 |
| §7 重量重心 | `Weight_cg_analysis.py` | 部品重量積み上げ → 重心 → 静安定・容積比 |
| §9 包絡線 | `flight_envelope.py` | 速度 vs 必要/利用可能 CL、失速速度 |
| §2 準備 | `screening.py` / `plot_polars.py` / `analysis_highres.py` | 翼型候補の NeuralFoil 一括スクリーニング |

新規スクリプトを作る場合も、**「役割・入力・出力」を冒頭の docstring に明記** してから実装する。

### 5.2 解析値の整合管理(★最重要)

計算書の数値と、それを生むスクリプト・実行結果は **`docs/analysis_results_v3.md`** に集約する(解析値の単一の真実)。計算書の数値を変更するときは、まず該当スクリプトを更新・再実行し、`analysis_results_v3.md` を更新してから計算書に反映する。この順序を守ることで、文書間および文書とコードの数値ドリフトを防ぐ。

**全章で統一する凍結値**(変更時は `analysis_results_v3.md` §5・§6 と整合を取ること):
- 機体総重量:設計値 36.5 g / 重量内訳の見積もり 36.1 g(§7、ブーム埋め込み接合反映後)
- 巡航速度 V = 7.5 m/s、動圧 q = 34.45 Pa
- 主翼 Cm = −0.162(VLM 値 −0.16238 の3桁丸め)、CFRP 引張強度 σ_ult = 1500 MPa、荷重倍率 n = 2
- 静安定:中立点 x_NP/c = 0.497、設計点 SM = 14.6 %、水平尾翼容積比 Vh = 0.552

**numpy 互換性**:numpy 2.x で `np.trapz` は削除済み。数値積分が必要な場合は `np.trapezoid` を使う。

## 7. 命名規則(2026-04-25 運用開始)

ファイル名は **「内容_条件_日付.拡張子」** が原則。**新規データ・図を作る前に必ず命名規則を確認・宣言してから着手すること。**

### XFLR5/XFoil 解析データ(`xfoil_data/`)
```
{翼型}_T{解析タイプ}_Re{Re値千分の一}_M{マッハ}_N{NCrit}.txt
例: AG36_T1_Re0.024_M0.00_N9.0.txt
```

### VLM スパン方向データ(`vlm_data/`)
```
spanwise_alpha{角度}deg_AR{AR値}_V{速度}ms_{日付}.txt
例: spanwise_alpha3deg_AR6_V7ms_2026-05-05.txt
```

### XFLR5 プロジェクト(`xflr5_projects/`)
```
{解析目的}_{日付}.xfl
例: AG36_VLM_mainwing_2026-05-05.xfl
```

### 生成プロット(Google Drive 側)
- フォルダ名:`{日付}_{解析テーマ}/`(例:`2026-05-05_vlm_mainwing_AG36/`)
- ファイル名:`{プロット内容}_{条件}.png`(例:`cl_distribution_alpha3deg_cruise.png`)

## 9. 典型的な依頼パターン

詳細は `requirements.md` §8 参照。よく来るのは:

1. **解析作業**:「XFLR5 の `vlm_data/spanwise_*.txt` を読んで Cl 分布をプロットして」
2. **計算書ドラフト**:「§3 尾翼の文章を書いて、根拠は `requirements.md` §4 から」
3. **検算**:「この応力計算、別の角度から検証して」
4. **資料整理**:「進捗報告書の §3 を簡潔に整えて」
5. **比較整理**:「A 案 / B 案 / C 案を比較して決定材料を整理して」

依頼を受けたら、まず関連する `requirements.md` / `progress_report_*.md` の該当セクションを確認してから着手する。

> 「設計部品が課題条件を満たすことを工学アプローチにより示す。その性能・機能を設計者が保証する。」

各章で必ず示す:
1. 力・モーメントの設定
2. 概略形状の設計(材料・加工法選定)
3. 強度設計(簡易モデル → 応力 → 安全率・許容応力)
4. 機構設計(取付方法、はめあい、公差)

**選定理由の比較検討は二次的**(主役は「条件を満たす保証」)。各章の選定理由は1〜2文で簡潔に。
