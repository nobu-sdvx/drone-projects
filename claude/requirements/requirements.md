## 0. 重要原則:公式要件 vs 設計目標の区別

| 種別 | 説明 | 凡例 | 変更可否 |
|---|---|---|---|
| 🟢 公式要件 | 課題PDF・POWERUP仕様から派生 | `official_requirements.md` 参照 | 不可(課題仕様変更時のみ) |
| 🟡 設計目標 | Nobu自身が根拠を持って設定 | このファイルの記載 | 設計の進展に応じて見直し可 |

**判断時の注意**:
- 計算結果が「目標」を満たさない場合、**目標の見直しが第一選択**(公式要件と区別する)
- 「目標達成のために設計を歪める」のは本末転倒
- 計算書では「公式要件をすべて満たしていること」を最優先で示し、「目標」については達成度を率直に評価

## 1. プロジェクト概要

### 1.1 プロジェクト名

応用機械設計製図Ⅰ ドローン班 ― 固定翼型ドローンの設計・製作

### 1.2 目的

POWERUP 4.0(モーター・コントローラ・バッテリー・受信機一体型)を流用し、機体・プロペラを自作した固定翼型ドローンを設計・製作する。AI解析ツール(NeuralFoil、XFLR5、XROTOR等)を活用し、設計根拠を定量的に示す計算書を作成する。

### 1.3 最終ゴール

8月上旬の屋内競技会で、個人成績とチーム成績の両方で上位入賞を目指す(🟡 自己目標)。

## 2. 公式要件のサマリー

詳細は `official_requirements.md` を参照。要点のみ:

🟢 **必須条件(違反不可、課題PDF)**:
- 機体形状: 曲面形状(折り紙NG、エッジNG)
- 衝突安全性: 自由落下・壁面衝突で破損なし
- 自作: 機体・プロペラ自作(POWERUP付属プロペラ流用不可)
- 揚力/推力装置: 翼+モータ駆動プロペラのみ
- 飛行環境: 屋内
- 予算: 9,000円(チーム3名標準)
- 推進無し滑空動画提出: 7/20まで

🟢 **POWERUP 4.0仕様(与件)**:
- 本体重量 19g
- 機体重量上限 20g(POWERUP仕様の動作保証範囲)
- 最大速度 約9 m/s
- 推力重量比 2:1(付属紙飛行機時)

🟢 **計算書の本質(課題PDF)**:
- 「設計部品が課題条件を満たすことを工学アプローチにより示し、設計者が保証する」
- 4項目: 力・モーメント設定 / 概略形状 / 強度設計(応力・安全率) / 機構設計

## 3. 設計目標(自己設定)

### 3.1 機体コンセプト 🟡

**矩形翼+十字尾翼構成**(シンプル設計)

**選定根拠**:
- 翼弦が一定のため2D翼型解析結果がそのまま3Dに適用できる
- 尾翼容積比という教科書的指標で設計根拠を示せる
- 発泡材の熱線カット成形との相性が良い
- AI解析ツール(XFLR5, NeuralFoil)との相性が最高

### 3.2 主要寸法目標 🟡

| 部位 | 寸法目標 | 根拠 |
|---|---|---|
| 翼幅 | 360 mm | POWERUP本体長233mmと釣り合い、AR=6を確保 |
| 翼弦 | 60 mm | 動作Re域と必要揚力の両立 |
| 翼面積 | 216 cm² | 翼幅×翼弦の派生 |
| アスペクト比 | 6.0 | 教科書範囲(片柳『飛行機設計入門』)、誘導抗力と構造のバランス |
| 水平尾翼 | 140 × 35 mm(49 cm²) | 尾翼容積比 Vh=0.55 から逆算 |
| 垂直尾翼 | 50mm高 × 40mm弦 | 尾翼容積比 Vv=0.04 から逆算 |
| 胴体全長 | 200 mm(POWERUP含め233mm) | POWERUPロッド長 |
| 翼面荷重 | 約1.5 kg/m² | W/S 教科書範囲 |

### 3.3 目標飛行性能 🟡

> **重要**:この節の数値は **すべて自己設定の目標値** であり、公式PDFには規定されていない。
> 計算結果が目標を満たさない場合は、目標値の見直しが第一選択。

| 項目 | 目標値 | 根拠 |
|---|---|---|
| 動作レイノルズ数 | 24,000〜36,000(巡航中心) / 20,000〜50,000(全域) | V × c / ν から逆算 |
| 巡航速度 | 6〜8 m/s | POWERUP最大9 m/sの70〜90% |
| 失速速度 | (算出される値、要件なし) | ~~4 m/s以下(誤って公式要件と認識していた、撤回)~~ |
| L/D(滑空比) | 8以上 | 屋内滑空に必要な最小限 |
| 必要CL(巡航) | 0.4〜0.7 | W/S=1.5、V=6〜8 m/s から逆算 |
| 最大CL(失速近傍) | 1.0以上 | 巡航余裕の確保 |

**失速速度の取り扱いに関する注記**(2026-05-05追加):
- 当初「失速速度4 m/s以下」を要件として記載していたが、公式PDF・POWERUP仕様にこの規定がないことを再確認
- 物理的にも、低Re域翼型のCL_max ≈ 0.9〜1.2 と本機の翼面荷重では4 m/s達成不可
- 計算書では「現設計での失速速度 ≈ 5 m/s、最大速度9 m/s に対し十分な飛行余裕」として記載予定
- ただし、構想発表(4/20)資料で言及している場合、計算書内で目標値見直しの一文を添える(今後確認)

### 3.4 重量配分目標 🟡 (2026-05-05改訂版)

> **改訂理由**:旧版(主翼2g)は **CFRPスパー約1.8gの計上漏れ** と推定。物理的根拠(発泡密度・CFRP断面・LE補強等)から再見積もりした結果、**現実的な機体重量は18g、総重量37g**。詳細根拠は調査レポート §H.1 および本日の構造解析スクリプト v2 を参照。

**主翼の内訳(根拠付き、5.0g)**:

| 部位 | 材料・寸法 | 計算 | 重量 |
|---|---|---|---|
| 翼本体 | EPS、AG36、360×60mm、密度25 kg/m³ | A=0.075×60²=270mm²、V=270×360=97cm³ | 2.4 g |
| メインスパー | CFRP φ2mm × 360mm、密度1.6 g/cm³ | V=π×0.1²×36=1.13cm³ | 1.8 g |
| 前縁補強 | バルサ 2×3mm × 360mm、密度0.15 g/cm³ | V=2.16cm³ | 0.3 g |
| 表面フィルム | Mylar等、両面≈432cm²、4 g/m² | 432×4/10⁴ | 0.2 g |
| 接着剤 | CA等、局所適用 | 推定 | 0.3 g |
| **主翼合計** | | | **5.0 g** |

**全体内訳**:

| 部品 | 旧target(2026-04-23) | 改訂target(2026-05-05) | 備考 |
|---|---|---|---|
| 主翼(発泡+スパー+補強+フィルム+接着) | 2 g | **5.0 g** | スパー漏れ補正 |
| 尾翼(発泡+補強+取付) | 1.5 g | **2.0 g** | レポート §H.1 範囲(2-3g)の下端 |
| 胴体フェアリング(POWERUPカーボンロッド利用) | 3 g | **3.0 g** | 据え置き |
| 自作プロペラ×2 + マウント(PLA) | 5 g | **5.0 g** | 据え置き |
| 接着剤・塗装・補強・コントロールリンク | 2 g | **3.0 g** | 上方修正 |
| **機体合計** | **13.5 g** | **18.0 g** | レポート §H.1範囲(12-17g)の上端寄り |
| POWERUP 4.0(🟢 公式仕様) | 19 g | **19 g** | 固定 |
| **総重量** | **32.5 g** | **37.0 g** | 🟢 公式上限39gとの余裕 2g |

**設計ポリシー**:
- 主シナリオ:**W=37g** で構造解析・性能評価を実施
- 公式上限39g を上回らない範囲での重量管理を実施(余裕 2g)
- 実機製作時に各部品を実測し、超過したら部品単位での軽量化を検討
- 構造側は φ2mm CFRP スパーで W=39g までの全範囲で σ_design < σ_allow を満たす(本日の感度解析で確認、SF=42)

## 4. 設計戦略

### 4.1 確定済みの設計判断 🟡

| 項目 | 確定事項 | 確定日 |
|---|---|---|
| 主翼翼型 | AG36 | 2026-05-05 |
| 翼平面形 | 矩形翼、AR=6、翼幅360mm、翼弦60mm | 2026-04-23(構想発表時) |
| 主構造材 | EPS発泡材 + φ2mmカーボンスパー(主翼) + バルサ前縁補強 | 2026-04-23 |
| 推進系 | POWERUP 4.0(🟢 与件) + 自作プロペラ | 2026-04-20 |

### 4.2 主翼解析確定値(2026-05-05、感度解析v2版) 🟡

| 項目 | W=32.5g(初版) | **W=37.0g(主)** | W=39.0g(上限) |
|---|---|---|---|
| 必要 CL_cruise | 0.49 | **0.56** | 0.59 |
| 巡航迎角 α | 3.2° | **4.1°** | 4.5° |
| 巡航 CL | 0.477(@α=3°) | **0.56(@α=4°近傍)** | 0.59(@α=4.5°近傍) |
| 巡航 CD | 0.045(@α=3°) | 0.050(推定) | 0.052(推定) |
| **巡航 L/D** | **10.5** | **約11.2** | **約11.3** |
| 巡航 Cm | -0.162 | -0.18(推定) | -0.19(推定) |
| 翼根曲げモーメント M_root | 11.71 mN·m | **13.33 mN·m** | 14.05 mN·m |
| 翼根曲げ応力 σ(n=2) | 29.8 MPa | **33.9 MPa** | 35.8 MPa |
| 安全率 SF(対 σ_ult=1500 MPa) | 50 | **44** | 42 |

**主シナリオ(W=37g)での結論**:
- 必要 CL=0.56 は AG36 の線形領域内(失速 α≈8° に対し巡航 α≈4°、4° 余裕)
- L/D=11.2 は目標 L/D≥8 をクリア(🟡 自己目標)
- 安全率 SF=44 は十分な余裕(SF≥10で過剰、軽量化余地あり)
- 全シナリオ(32.5g〜39g)で構造的に成立(本日の感度解析で確認)

**注意**:
- 上記は α=3°でのXFLR5計算値を線形CL-α関係で外挿。厳密にはW=37g時の翼根曲げモーメントは **再度 XFLR5 で α=4° 近傍を解析して確定** すべき(時間あれば 5/6 タスクに追加)
- Re=24kは外挿領域(UIUC実測Re下限60k、Drela polar下限50k)。絶対値に±20%の不確実性を見込むこと(調査レポート §F.3 警告)

## 6. 設計優先順位

### 6.1 🟢 守らねばならない順(公式要件優先)

1. **総重量40g以下**(POWERUP仕様、これを超えるとそもそも飛ばない)
2. **衝突無破損**(課題PDF、自由落下/壁面衝突で機体・床面に損傷なし)
3. **機体形状(曲面)**(課題PDF、折り紙・エッジ平板NG)
4. **動力なし滑空可能**(課題PDF、7/20までに動画提出)
5. **計算書での工学的妥当性**(課題PDF、これが80%評価)

### 6.2 🟡 自己目標(公式要件を満たした上での最適化)

1. 競技会上位入賞(直進+旋回の合計ポイント最大化)
2. L/D ≥ 8(滑空性能、🟢 動力なし滑空可能要件の実現手段として)
3. 軽量化(製作精度向上、衝突時被害軽減)
4. 計算書のクオリティ向上(AI解析ツール多重検証等)

## 7. AI使用方針

- 設計判断はAIではなくNobu自身が行う(計算書での説明責任)
- AIは「解析の高速化」「文書化補助」「知識検索」「検算」に使う
- 設計の重要判断はrequirements.md(本ファイル)とofficial_requirements.mdに残す
- 過去の判断と根拠を進捗報告書(`progress_report_YYYY-MM-DD.md`)に記録

## 8. 想定される依頼パターン

AIに作業を依頼する際の典型例:

1. **解析作業依頼**:「XFLR5の.txtを読んでCl分布をプロットするスクリプトを書いて」
2. **計算書セクションのドラフト**:「§2主翼の概略設計の文章を書いて、根拠は本ファイルから」
3. **検算**:「この応力計算、別の角度から検証して」
4. **資料整理**:「進捗報告書の〇〇章をブラッシュアップして」
5. **判断材料の整理**:「A案/B案/C案を比較して決定材料を整理して」

## 11. 補足:このドキュメントの使い方

1. **AIに依頼する前**に、`official_requirements.md` と本ファイルを併せてコンテキストとして貼り付ける
2. **具体的な依頼**を続けて書く(§8のパターンを参考に)
3. AIからの回答を計算書・図面・スクリプトに反映
4. 設計の進展に応じて、本ドキュメントの「3. 設計目標」「4. 設計戦略」を更新(履歴は §10に追記)
5. 公式要件側に変更があった場合のみ、`official_requirements.md` を更新

**両ドキュメントを最新状態に保つことで、AIとの連携効率と判断ミス防止の両立が図れる。**
